🔧 Technical Notes: Document Retrieval Methods

Date: October 22, 2025 | Issue: web_fetch tool truncation | Status: WORKAROUND IDENTIFIED

⚠️ CRITICAL ISSUE: The web_fetch tool truncates large HTML documents, causing session start protocol failures. This document provides the permanent workaround and prevention strategies.

The Problem

Issue: The web_fetch tool has an undocumented limitation where it truncates large HTML documents during retrieval.

Specific Failure Case

When fetching the Progress Diary at https://chaver.com/torah-commentary-project/Admin/ProgressDiary.html:

  • Expected: Full HTML document with all diary entries (482+ lines)
  • Actual: Truncated HTML showing only the first entry (October 19, 2025)
  • Missing: October 20 and October 22 entries completely absent
  • File size: Document exceeds ~50KB threshold where truncation occurs

Impact on Workflow

💥 Token Cost: ~40,000 Tokens Wasted

This issue caused multiple debugging sessions across several days, consuming significant token budget before the root cause was identified.

  1. Session start fails - Claude cannot see what was accomplished in previous sessions
  2. Continuity breaks - No knowledge of last session's work, decisions, or next priorities
  3. User frustration - System appears "useless" when it can't retrieve current state
  4. Wasted tokens - Multiple failed attempts to fetch the same document
  5. Lost work context - Must manually explain previous session's accomplishments

The Solution

Use bash_tool with curl instead of web_fetch for large documents.

Why This Works

  • curl retrieves the complete document without truncation
  • File system operations are reliable and predictable
  • Can extract specific sections using standard Unix tools
  • No token limit on bash tool operations
  • Provides complete control over what data is retrieved and processed

Step-by-Step Implementation

Step 1: Download Complete Document

# Download complete diary to temporary location
curl -s "https://chaver.com/torah-commentary-project/Admin/ProgressDiary.html" > /tmp/diary.html

Step 2: Find All Entries

# Locate all entry markers with line numbers
grep -n "entry-date\|<h3 style=\"color: #2c3e50" /tmp/diary.html

This shows line numbers for all entries, with most recent first.

Step 3: Extract Most Recent Entry

# Extract specific line range (adjust numbers based on grep output)
sed -n '234,300p' /tmp/diary.html

Step 4: Parse Key Information

Extract and present to user:

  • Session date
  • Work completed
  • Files modified
  • Next priorities
  • Outstanding issues

Document Size Guidelines

Safe to Use web_fetch

Document Type Typical Size Status
Admin Hub (index.html) < 20KB ✅ Safe
Session Start Template < 30KB ✅ Safe
Individual unit commentaries < 40KB ✅ Safe
CSS files < 20KB ✅ Safe

Use bash_tool Instead

Document Type Typical Size Status
Progress Diary > 50KB (grows over time) ⚠️ Confirmed issue
Database documentation Variable ⚠️ May have issues
Complete protocol documents > 50KB ⚠️ Monitor
Any file that grows over time Variable ⚠️ Use bash_tool

Long-Term Fix Options

Option 1: Pagination (Recommended)

Split Progress Diary into multiple files:

  • ProgressDiary-Current.html - Last 3-5 entries only (< 30KB)
  • ProgressDiary-Archive.html - All older entries

Pros: Session start protocol can use web_fetch reliably

Cons: Requires periodic manual archiving

Option 2: JSON API

Create a lightweight endpoint that returns diary data:

  • Returns only last N entries as JSON
  • Much smaller payload than full HTML
  • Faster parsing and processing

Pros: Automated, efficient, scalable

Cons: Requires backend development

Option 3: Summary File

Maintain a separate lightweight file:

  • ProgressDiary-LastSession.txt
  • Contains only most recent entry
  • Plain text format (< 5KB)
  • Always fetchable with web_fetch

Pros: Simple, reliable, no code changes

Cons: Requires updating two files per session

Prevention Strategies

Going Forward

  1. Monitor file sizes - If any admin document exceeds 50KB, consider splitting
  2. Test retrieval - Verify web_fetch works for new documents before relying on them
  3. Have fallback - Always provide bash_tool alternative in protocol documentation
  4. Document limits - Note which files need special handling
  5. Regular archiving - Move old diary entries to archive file quarterly

Testing Checklist

  • Most recent entry is correctly identified
  • All key information extracted (date, work, priorities)
  • Token usage reasonable (< 5% for session start)
  • Process is documented in protocol
